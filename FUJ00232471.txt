process into the following sub-processes: (a) Manage the
release, (b) Develop the release and (c) Test the release.

divides Software Release Management into the following sub-processes:
(a) I Manage the release
(b) I Develop the release
(c) Test the release

Subsequent versions of this document will include additional levels of
decomposition.

2 Scope

This document includes software releases that are initiated by PinICLs and
CPs.

3.2 Manage the Release
Description 1. A live PinICL is received from the Manage Live Incident

process and scheduled for development.

2. An approved CP is received from the Manage Change)
process and scheduled for development.

3. Items in Draft Counter News will be passed to the Manage
the Release process if they are considered to be relevant.

4. CSRM return their comments to the Manage CommunicationI
with PON process.

5. If it is a major release, then CSRM send words for a Counter
News Item to PON.

6. CSRM monitor progress of releases. CSRM send a WeekI:
Schedule and a Daily Notification of Releases to PON.

7. CSRM authorises Development Directorate to start work on)
the PinICL or CP.

8. Notification of software development progress and
completion is received from the Development Directorate.

9. For fast-track releases CSRM send a ‘Start Testing Trigger’
to Test the Release process when the item is ready to be
tested.

Note: For major releases, the Development Directorate sends}
a ‘Start Testing Trigger’ to Test the Release process.

10.CSRM receive details of progress from Test the Release)
and adjust schedules accordingly.

11.CSRM send to Test the Release changes to priorities so that
they can adjust testing.

112.CSRM decide whether the release can be released to live. If
they decide to release it, they send a Release Note to
Distribute the Release (ISD).

13.A closed Release Note is received by CSRM when the)
release has been installed in the live environment.

3.3. Develop the Release
Description 1. Development Directorate receives PinICLs and CPs that

require to be developed.

2. When appropriate Development Directorate obtains the
correct version of software to be developed from Provide
Safe Storage for Software (Configuration Management).

3. Development Directorate carries out changes in accordance)
with the PinICLs and CPs.

4. Development Directorate provides a Notification of
Completion to CSRM when the development is finished (Fast
track).

5. Development Directorate provides the developed software)
back to Provide Safe Storage for Software (Configuration)
Management).

6. Development Directorate send a ‘Start Testing Trigger’ to)
Test the Release (Major Release).
